# 200万汉字输出能力评估报告

> 评估日期：2026-06-06 | 目标：判断 novel-ai-writer 是否能输出一篇完整的 200 万汉字小说

---

## 一、总评

**结论：当前系统架构可以输出 200 万字，但存在 7 个实际问题会导致中途崩溃或质量崩塌。逐个修复后可达标。**

| 维度 | 状态 | 瓶颈级别 |
|------|------|----------|
| 单章生成 | ✅ 可用 | — |
| 逐章循环 | ✅ 可用（手动触发） | — |

## 二、逐层分析

### 2.1 单章生成能力 ✅

```
CHAPTER_SYSTEM (约700行) + buildExecutionConstraints (副零层铁律)
→ 每章输出 2000-5000 字（默认 estimated_words: 3000）
→ max_tokens: 16384（约等于中文 8000-12000 字输出上限）
→ 实际限制：单章最多 ~10000 字
```

**结论**：单章生成能力强，但受 `max_tokens=16384` 限制，单次 API 调用最多输出约 8000-12000 汉字。如果用户要单章 5000+ 字，可能需要续写。

### 2.2 逐章循环 ✅

当前系统逐章手动生成：
1. 生成细纲（为整卷规划每章情节点）
2. 逐章点击生成 → AI 写正文 → 自动触发记录官/事件提取/一致性检查
3. 每章生成后自动更新 `context_state`（角色状态+情节摘要）

**结论**：逐章生成逻辑健壮，但需要人工每次点击"生成"按钮。

### 2.3 上下文窗口 ⚠️🟡

```
实际注入到每章 user prompt 的内容：
├── CHAPTER_USER 模板本身：~200 tokens
├── stateContext（叙事状态）：~300 tokens
├── styleContext（风格库 V4 22类，截断2000字）：~1000 tokens
├── personalityContext（人格库 V2 5核，截断1500字）：~800 tokens
├── canonFactsContext（事实簿硬规则）：~500-2000 tokens（随章节增长）
├── plotSummary（前情摘要，截断500字）：~250 tokens
├── prevExcerpt（上章结尾，800字）：~400 tokens
├── disassemblyContext（拆文库，截断1200字）：~600 tokens
├── plot_beats（情节点序列 8-15条）：~400 tokens
├── infoPermissionContext（信息权限）：~200 tokens
├── timeProgressionContext（时间推移）：~300 tokens
├── consistencyChecklist（一致性检查）：~200 tokens
├── buildMinimalContext（角色+世界上下文）：~500 tokens
├── CHAPTER_SYSTEM（system prompt 700行）：~3500 tokens
├── buildExecutionConstraints（副零层铁律）：~1500 tokens
└── 其他（volContext, nodeContext, hook 等）：~400 tokens
────────────────────────────────────────────────
估算总计：~10000-14000 tokens（含 system prompt）
```

**关键约束**：
- `max_tokens: 16384` 仅限制**输出**，不限制输入
- DeepSeek v4 上下文窗口：128K tokens（约 6-8 万汉字输入）
- 当前每章注入约 1-1.4 万 tokens → 在 128K 窗口内游刃有余

**但注意**：如果 `canon_facts` 表积累了数百条事实，`canonFactsContext` 可能膨胀到 5000+ tokens。当前代码没有对 canonFactsContext 做截断。

### 2.4 叙事一致性（长篇小说最大瓶颈）🔴

这是当前系统最严重的问题。200 章后，以下问题必然出现：

#### 2.4.1 角色状态漂移
- `context_state` 表只存一个全局的 `character_state` JSON 对象
- 200 章后这个 JSON 会变得极其庞大（所有角色的所有状态变化）
- `CONTEXT_UPDATE_USER` 只传入 `prevCharState` 和 `newContent.slice(0, 5000)`
- AI 每次更新时只看到上一次的全局状态 + 新章节 → **不知道 100 章前的角色状态**

#### 2.4.2 情节遗忘
- `plotSummary` 是不断被覆盖的单一字符串
- 没有分卷/分阶段的情节摘要存档
- 第 150 章生成时，AI 不知道第 20 章发生过什么关键事件

#### 2.4.3 伏笔管理
- `foreshadowing_registry` 表存在，但只记录 `status='active'` 的伏笔
- 回收后标记为 `recovered`，但回收质量无人检查
- 没有"已埋未回收超 30 章"的预警

#### 2.4.4 时间线一致性
- `story_timeline` 表按 `absolute_day` 记录，理论上可以追踪
- 但生成第 150 章时，`timeProgressionContext` 只计算了"距今天数"
- 没有检查"第 20 章说怀孕 3 个月，第 80 章过去了 30 天却说孩子 1 岁了"这类逻辑矛盾

#### 2.4.5 跨卷一致性
- 每卷的 `volume_forbidden` / `emotion_stage` / `global_info_quota` 刚补齐
- 但没有跨卷检查：卷 3 可能揭露了卷 1 禁止揭露的内容

### 2.5 数据库容量 ⚠️🟡

```
SQLite 最大数据库大小：~281 TB（理论值）
单章存储：
  - chapters.content：~3000 字 ≈ 9 KB（UTF-8）
  - chapter_summaries：~500 字 ≈ 1.5 KB
  - canon_facts 增量：~1-2 条/章
  - canon_events：~1-3 条/章
  - foreshadowing_registry 增量：~0.5 条/章
  - story_timeline 增量：~0.5 条/章
  - character_arc_log 增量：~0.3 条/章
  - relationship_timeline 增量：~0.2 条/章

200万字 ≈ 667章 × 3000字/章
数据库估算：
  chapters 表：667 × 9KB ≈ 6 MB
  其他表合计：~10 MB
  总数据库：~15-20 MB
```

**结论**：SQLite 完全够用，20MB 数据库对 sql.js（内存数据库）也是安全的。

### 2.6 AI API 成本 ⚠️🟡

```
估算参数：
- 200万字 = 667章 × 3000字/章
- 每章生成：1 次正文 API + 1 次记录官 API + 1 次事件提取 API + 1 次上下文更新 API = 4 次 API 调用
- 细纲生成：每卷 1 次（假设 8 卷）= 8 次
- 卷纲生成：每卷 1 次 = 8 次
- 大纲生成：1 次

总 API 调用估算：667×4 + 8 + 8 + 1 = 2685 次

每章 token 消耗估算：
- 正文生成：input ~12000 tokens + output ~2000 tokens = ~14000 tokens
- 记录官：input ~3000 + output ~300 = ~3300 tokens
- 事件提取：input ~3000 + output ~300 = ~3300 tokens
- 上下文更新：input ~2000 + output ~200 = ~2200 tokens
- 每章小计：~22800 tokens

总 token 消耗：667 × 22800 + 17 × 15000 ≈ 1520 万 + 25 万 ≈ 1545 万 tokens

DeepSeek v4 价格（2026年）：
- 输入：约 ¥1/百万 tokens
- 输出：约 ¥2/百万 tokens
- 假设输入:输出 ≈ 5:1
- 估算总费用：约 ¥20-30
```

**结论**：API 成本很低（远低于雇人写），但调用次数多（2685 次），如果每章人工点击会非常耗时。

### 2.7 人力操作成本 🔴

```
当前工作流（完全手动）：
1. 生成大纲 → 点击
2. 生成卷纲（每卷一次） → 点击
3. 生成细纲（每卷一次） → 点击
4. 逐章生成正文（667次） → 每次点击
5. 每章查看违规标记 → 可选
6. 修改违规内容 → 可选

667 章 × 每章等待 ~30-60 秒（AI 生成 + 记录官 + 检查）= 5.5-11 小时纯等待
加上人工操作时间：可能需要 10-20 小时
```

**核心缺失**：没有"自动逐章生成"模式。当前系统只支持手动一章一章点。

### 2.8 全局情节连贯性 🔴

当前一致性检查（`checkChapter`）只检查**单章内**的问题：
- 禁词检查
- 人称一致
- 信息泄露（与 `revealed_level` 比对）

但以下检查完全缺失：
- **角色弧线检查**：主角从第 1 章到第 667 章的变化是否连贯
- **情节逻辑检查**：事件 A 导致 B 导致 C 的因果链是否完整
- **节奏检查**：高潮低谷交替是否均匀（只在细纲层面设计了，正文未验证）
- **字数分布检查**：是否某卷过长某卷过短
- **伏笔回收率**：埋了多少收了多少

## 三、要输出 200 万字需要修复什么

### 🔴 P0（不修复无法完成）

1. **自动逐章生成模式**
   - 添加"自动生成全部章节"按钮
   - 自动依次生成 → 自动记录官 → 自动事件提取 → 自动下一章
   - 可暂停/恢复
   - 预估工作量：3-5 小时

2. **上下文窗口管理**
   - `canonFactsContext` 需要分级截断：全量硬规则 + 摘要软规则
   - `context_state.plot_summary` 需要分卷存档，不要只保留一个全局摘要
   - 预估工作量：2-3 小时

### 🟡 P1（不修复质量会下降）

3. **跨卷一致性检查**
   - 生成第 N 卷前，检查前 N-1 卷的 `volume_forbidden` 是否被后续违反
   - 生成第 N 章前，注入前 N-1 章的关键事件摘要（不是全局摘要，是分阶段摘要）
   - 预估工作量：4-6 小时

4. **伏笔回收追踪**
   - 每 30 章检查一次未回收伏笔
   - 对超过 50 章未回收的伏笔发出警告
   - 预估工作量：1-2 小时

5. **角色状态分版本存储**
   - 不要只存一个全局 `character_state`
   - 每章/每 5 章存一份角色状态快照
   - 预估工作量：2-3 小时

### 🟢 P2（锦上添花）

6. **全局情节连贯性检查**
   - 全篇生成完毕后，用 AI 检查因果链完整性
   - 检测"突然出现""莫名消失"的角色
   - 预估工作量：4-8 小时

7. **批量生成后的整体润色**
   - 统一人称、修正不一致的细节
   - 统一格式（段落、对话风格）
   - 预估工作量：3-5 小时

## 四、字数计算验证

```
200万汉字 = 2,000,000 字

假设配置：
- 每章 3000 字（estimated_words 默认值）
- 需要 667 章

假设卷结构：
- 每卷 80-100 章
- 约 7-8 卷

每卷字数：
- 80 章 × 3000 字 = 24 万字/卷
- 8 卷 × 24 万字 = 192 万字（接近 200 万）

调整方案：
- 每章 4000 字 × 500 章 = 200 万字
- 每章 3000 字 × 667 章 = 200 万字
- 每章 2500 字 × 800 章 = 200 万字
```

**结论**：从计算上完全可行。当前系统默认每章 3000 字，667 章即可。

## 五、最终结论

| 问题 | 能绕过去吗 | 修复难度 |
|------|-----------|---------|
| 没有自动续写 | ❌ 不能，667 次手动点击不可行 | 中 |
| 上下文膨胀 | ⚠️ 可以但第 200 章后质量下降 | 低 |
| 叙事遗忘 | ❌ 200 章后 AI 不知道前面发生了什么 | 中 |
| 伏笔漂移 | ⚠️ 可以但需人工盯着 | 低 |
| 跨卷矛盾 | ⚠️ 可以但需人工校对 | 中 |
| API 成本 | ✅ 很便宜（~¥30） | — |
| 数据库容量 | ✅ 完全够用（~20MB） | — |

**一句话总结**：当前系统能写出 200 万字的**初稿**，但需要一个人在旁边不停地点击"生成下一章"按钮，而且第 200 章之后 AI 会忘记前面发生过什么。修复 P0+P1 共 6 项后，可以基本自动化地输出 200 万字且质量可接受的长篇小说。
